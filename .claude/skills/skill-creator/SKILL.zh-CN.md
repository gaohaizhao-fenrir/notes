# Skill Creator（技能创作器）

用于创建新技能并迭代改进的技能。

从宏观上看，创建技能的流程大致如下：

- 想清楚希望技能做什么、大致怎么做
- 写出技能草稿
- 准备若干测试提示词，在有权限访问该技能的情况下用 Claude 跑一遍
- 帮助用户从定性与定量两方面评估结果
  - 在任务后台运行时，若还没有定量评测，就起草一些；若已有，可按需沿用或修改。然后向用户解释这些评测（若原本就有，则解释已有的）
  - 使用 `eval-viewer/generate_review.py` 脚本向用户展示结果供查看，并展示定量指标
- 根据用户对结果的评估反馈（以及定量基准中暴露的明显问题）重写技能
- 重复直到满意
- 扩大测试集，在更大规模上再试一次

使用本技能时，你的工作是判断用户处于流程的哪一步，并协助推进。例如用户说「我想做一个关于 X 的技能」，你可以帮助收窄含义、写草稿、写测试用例、弄清他们希望如何评估、跑完所有提示词并循环。

另一方面，若用户已有技能草稿，可以直接进入评测/迭代环节。

当然要保持灵活：若用户说「不想跑一大堆评测，随便聊聊就行」，也可以那样做。

技能完成后（顺序仍可灵活调整），还可以运行专门的「技能描述优化」脚本，以改善技能的触发效果。

明白了吗？明白就好。

## 与用户沟通

技能创作器的使用者对编程术语的熟悉程度差异很大。你可能还没听说（也正常，这是最近才有的现象）：Claude 的能力正让水管工也打开终端、让长辈去搜「怎么装 npm」。但多数用户可能仍具备一定计算机素养。

请注意上下文线索，决定如何措辞。默认情况下可参考：

- 「evaluation」「benchmark」略偏术语，但一般可以接受
- 对「JSON」「assertion」等词，最好先确认用户明显了解再直接使用，否则应附带简短说明

若有疑虑可简短解释术语；不确定用户是否理解时，可用一两句定义澄清。

## 创建技能

### 捕捉意图

先理解用户意图。当前对话里可能已经包含用户想固化的工作流（例如他们说「把这个做成技能」）。若是如此，先从对话历史里抽取答案——用过的工具、步骤顺序、用户纠正过的点、观察到的输入/输出格式。用户可能需要补全信息，进入下一步前应确认。

1. 这个技能应让 Claude 能做什么？
2. 何时应触发？（用户可能怎么说、在什么情境下）
3. 期望的输出格式是什么？
4. 是否建立测试用例以验证技能？对可客观验证的输出（文件转换、数据抽取、代码生成、固定工作流步骤），测试用例很有用；对主观输出（文风、美术），往往不需要。根据技能类型建议默认选项，由用户决定。

### 访谈与调研

主动追问边界情况、输入/输出格式、示例文件、成功标准、依赖等。把这部分理清后再写测试提示词。

检查可用的 MCP——若有助于调研（搜文档、找类似技能、查最佳实践），有子代理则并行调研，否则内联进行。带着背景去谈，减轻用户负担。

### 撰写 SKILL.md

根据访谈结果补全以下部分：

- **name**：技能标识符
- **description**：何时触发、做什么。这是主要的触发机制——既要写技能做什么，也要写具体在哪些情境下使用。所有「何时使用」的信息都放在这里，不要只写在正文里。注意：目前 Claude 容易「欠触发」——该用技能时没用。为缓解这一点，描述可以略「主动」一些。例如，不要只写「如何搭建一个简单的快速仪表盘展示内部 Anthropic 数据」，可以写成「如何搭建……。只要用户提到仪表盘、数据可视化、内部指标，或希望展示任何公司内部数据，即使没明说『仪表盘』，也应使用本技能。」
- **compatibility**：所需工具、依赖（可选，很少需要）
- **技能的其余部分 :)**

### 技能撰写指南

#### 技能的结构

```
skill-name/
├── SKILL.md（必需）
│   ├── YAML 前置元数据（name、description 必填）
│   └── Markdown 说明
└── 打包资源（可选）
    ├── scripts/    - 可执行代码，用于确定性/重复性任务
    ├── references/ - 按需载入上下文的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体等）
```

#### 渐进式披露

技能采用三级加载：

1. **元数据**（name + description）——始终在上下文中（约 100 词）
2. **SKILL.md 正文**——技能触发时在上下文中（理想情况下少于 500 行）
3. **打包资源**——按需（无上限；脚本可在不整文件载入的情况下执行）

上述字数仅为参考，必要时可以更长。

**关键模式：**

- 尽量让 SKILL.md 少于 500 行；若接近上限，增加一层结构，并清楚指明使用技能的模型接下来应去哪里跟进。
- 在 SKILL.md 中明确引用文件，并说明何时阅读
- 对较大的参考文件（>300 行），附上目录

**领域组织**：若技能覆盖多个领域/框架，按变体组织：

```
cloud-deploy/
├── SKILL.md（工作流 + 选择逻辑）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

Claude 只读相关的参考文件。

#### 不令人意外原则

不言而喻：技能不得包含恶意软件、利用代码或任何可能危害系统安全的内容。技能的意图不应与用户从描述中理解到的相悖。不要配合创建误导性技能，或用于未授权访问、数据外泄等恶意目的。「扮演某某角色」之类则通常可以。

#### 写作模式

说明优先使用祈使语气。

**定义输出格式**——可以像这样：

```markdown
## 报告结构
始终使用以下模板：
# [标题]
## 执行摘要
## 主要发现
## 建议
```

**示例模式**——包含示例很有用，可如下格式化（若示例里已有「输入」「输出」字样，格式可略作调整）：

```markdown
## 提交信息格式
**示例 1：**
输入：新增用户认证，使用 JWT
输出：feat(auth): implement JWT-based authentication
```

### 文风

尽量向模型解释「为什么重要」，而不是堆砌冷冰冰的 MUST。运用心智理论，让技能通用而非绑死在个别示例上。先写一版草稿，再换眼光通读改进。

### 测试用例

写完技能草稿后，想 2～3 个真实的测试提示词——要像真实用户会说的话。与用户分享：[不必逐字使用]「我想先试这几个测试用例，你看是否合适，要不要再加？」然后运行它们。

把测试用例存到 `evals/evals.json`。先不要写断言——只写提示词。等运行进行时再在下一步起草断言。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示",
      "expected_output": "对期望结果的描述",
      "files": []
    }
  ]
}
```

完整 schema（含后续要加的 `assertions` 字段）见 `references/schemas.md`。

## 运行与评估测试用例

本节是一条连续流程——不要中途停下。不要使用 `/skill-test` 或其他测试类技能。

把结果放在 `<skill-name>-workspace/`，与技能目录同级。在工作区内按迭代组织（`iteration-1/`、`iteration-2/` 等），每次迭代里每个测试用例一个目录（`eval-0/`、`eval-1/` 等）。不要一次性建好全部目录——用到再建。

### 第一步：在同一轮中启动所有运行（含技能与基线）

对每个测试用例，在同一轮里派生两个子代理——一个带技能，一个不带。这一点很重要：不要先跑带技能的再回头补基线。尽量同时启动，使完成时间接近。

**带技能的运行：**

```
执行此任务：
- 技能路径：<path-to-skill>
- 任务：<eval prompt>
- 输入文件：<eval 若有文件则列出，否则写 "none">
- 输出保存到：<workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 需要保存的输出：<用户关心的内容——例如「最终的 .docx」「最终 CSV」>
```

**基线运行**（相同提示词，基线含义依情境而定）：

- **创建新技能**：完全不带技能。相同提示词，不指定技能路径，保存到 `without_skill/outputs/`。
- **改进已有技能**：使用旧版本。编辑前先把技能快照（`cp -r <skill-path> <workspace>/skill-snapshot/`），再让基线子代理指向该快照。保存到 `old_skill/outputs/`。

为每个测试用例写 `eval_metadata.json`（断言可先为空）。每个 eval 起一个能说明测什么的描述性名称——不要只叫「eval-0」。目录名也用这个名字。若本迭代使用了新的或改过的评测提示，要为每个新 eval 目录创建这些文件——不要假设会从上一轮继承。

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "用户的任务提示",
  "assertions": []
}
```

### 第二步：在运行进行时起草断言

不要干等运行结束——这段时间可以起草每个测试用例的定量断言并向用户解释。若 `evals/evals.json` 里已有断言，审阅并向用户说明其检查内容。

好的断言应可客观验证，名称要有描述性——在基准查看器中一眼能看懂每条在验什么。偏主观的技能（文风、设计质量）更适合定性评估——不要强行给需要人类判断的内容加断言。

断言定稿后，更新各 `eval_metadata.json` 和 `evals/evals.json`。同时向用户说明在查看器中会看到什么——既有定性输出，也有定量基准。

### 第三步：运行完成时记录耗时数据

每个子代理任务完成时，你会收到包含 `total_tokens` 和 `duration_ms` 的通知。请立即把数据写入该运行目录下的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是唯一能捕获该数据的时机——它来自任务通知，不会在其他地方持久化。收到通知就逐条处理，不要攒批。

### 第四步：评分、汇总并启动查看器

全部运行结束后：

1. **为每次运行评分**——派生评分子代理（或内联评分），阅读 `agents/grader.md`，按断言检查输出。结果保存到各运行目录的 `grading.json`。`grading.json` 的 expectations 数组必须使用字段 `text`、`passed`、`evidence`（不要用 `name`/`met`/`details` 等变体）——查看器依赖这些确切字段名。对可用程序检查的断言，编写并运行脚本，而不是凭目测——脚本更快、更可靠，且可跨迭代复用。

2. **汇总为基准**——在 skill-creator 目录下运行汇总脚本：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   会生成 `benchmark.json` 和 `benchmark.md`，包含各配置的通过率、耗时与 token，以及均值 ± 标准差与差值。若手动生成 `benchmark.json`，查看器期望的 schema 见 `references/schemas.md`。每个 `with_skill` 版本应放在对应基线之前。

3. **做一次分析**——阅读基准数据，找出汇总统计可能掩盖的模式。见 `agents/analyzer.md`（「Analyzing Benchmark Results」一节）应关注什么——例如无论技能如何都通过的断言（无区分度）、方差很大的评测（可能不稳定）、时间与 token 的权衡等。

4. **启动查看器**，同时展示定性输出与定量数据：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   从第 2 轮迭代起，还要传入 `--previous-workspace <workspace>/iteration-<N-1>`。

   **Cowork / 无界面环境：** 若 `webbrowser.open()` 不可用或环境无显示器，使用 `--static <output_path>` 生成独立 HTML，而不是启动服务器。用户点击「Submit All Reviews」时，反馈会以 `feedback.json` 文件形式下载。下载后把 `feedback.json` 复制到工作区目录，供下一轮迭代使用。

注意：请用 `generate_review.py` 生成查看器，无需手写 HTML。

5. **告诉用户**大意如下：「已在浏览器中打开结果。有两个标签页——『Outputs』可逐条查看测试用例并留言反馈，『Benchmark』展示定量对比。你看完后回到这里告诉我。」

### 用户在查看器中看到的内容

「Outputs」标签一次展示一个测试用例：

- **Prompt**：给出的任务
- **Output**：技能产出的文件，尽可能内联渲染
- **Previous Output**（第 2 轮起）：折叠区，显示上一轮输出
- **Formal Grades**（若已评分）：折叠区，显示各断言通过/失败
- **Feedback**：随输入自动保存的文本框
- **Previous Feedback**（第 2 轮起）：上次留言，显示在文本框下方

「Benchmark」标签展示统计摘要：各配置的通过率、耗时与 token 用量，含逐评测细分与分析观察。

用上一项/下一项按钮或方向键导航。完成后点击「Submit All Reviews」，将所有反馈保存到 `feedback.json`。

### 第五步：阅读反馈

用户表示完成后，阅读 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少坐标轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完美，很喜欢", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈表示用户认为没问题。改进时优先处理用户有具体意见的测试用例。

用完后可结束查看器进程：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是循环的核心。你已运行测试用例，用户已审阅结果，现在要根据反馈把技能做得更好。

### 如何思考改进

1. **从反馈中泛化。** 这里的大图景是：我们希望技能能被使用无数次（字面意义或更多）并覆盖大量不同提示词。你和用户只在少数样例上反复迭代，是为了加快迭代速度；用户对这些样例很熟悉，能快速评估新输出。但若你和用户共同开发的技能只适用于这些例子，就没有价值。与其做细碎、易过拟合的改动，或压迫性的 MUST，若某问题很顽固，可以尝试换隐喻、推荐不同的工作方式。尝试成本相对较低，或许能撞到更好的方案。

2. **保持提示精简。** 删掉没有贡献的内容。务必阅读对话记录，不要只看最终输出——若技能似乎在让模型浪费时间做无用功，可尝试删掉导致该行为的段落，观察效果。

3. **解释原因。** 努力解释你要求模型做的每件事背后的**原因**。当今的 LLM 很聪明，具备不错的心智理论；在好的约束下，它们能超越死记硬背，真正把事情做成。即使用户反馈简短或带着情绪，也要努力理解任务、用户为什么这么说、以及用户实际写了什么，并把这种理解写进说明。若发现自己在大写 ALWAYS 或 NEVER，或结构过于僵硬，这是黄旗——如可能，改写成解释理由，让模型理解「为什么重要」。这更人性化、更有力、也更有效。

4. **寻找跨测试用例的重复劳动。** 阅读测试运行的记录，注意子代理是否各自写了类似的辅助脚本，或对某件事采取了相同的多步做法。若三个测试用例都让子代理写了 `create_docx.py` 或 `build_chart.py`，这是强烈信号：技能应打包该脚本。写一次，放进 `scripts/`，并在技能中说明使用方式。这样以后的每次调用都不必重复造轮子。

这件事很重要（我们试图创造巨大的经济价值），思考时间通常不是瓶颈；请花时间琢磨。建议先写一版修订稿，再重新审视并改进。真正站在用户角度，理解他们想要什么、需要什么。

### 迭代循环

改进技能之后：

1. 将改进应用到技能
2. 在新目录 `iteration-<N+1>/` 中重新运行全部测试用例，含基线。若你在创建新技能，基线始终是 `without_skill`（无技能）——各轮迭代保持不变。若在改进已有技能，由你判断基线用用户最初版本还是上一轮。
3. 启动审阅工具时传入 `--previous-workspace` 指向前一轮工作区
4. 等用户审阅并告知完成
5. 阅读新反馈，再次改进，重复

持续到：

- 用户表示满意
- 反馈全空（看起来都好）
- 已无明显进展

## 进阶：盲测对比

若需要对技能的两个版本做更严格对比（例如用户问「新版本真的更好吗？」），可使用盲测对比系统。细节见 `agents/comparator.md` 和 `agents/analyzer.md`。基本思路是：把两份输出交给独立代理，不告知哪份来自哪个版本，由其评判质量，再分析胜者为何胜出。

这是可选项，需要子代理，多数用户用不上。通常人工审阅循环已足够。

## 描述优化

SKILL.md 前置元数据中的 `description` 字段是决定 Claude 是否调用技能的主要机制。创建或改进技能后，可主动提出优化描述以提升触发准确率。

### 第一步：生成触发评测查询

创建 20 条评测查询——混合「应触发」与「不应触发」。保存为 JSON：

```json
[
  {"query": "用户提示", "should_trigger": true},
  {"query": "另一条提示", "should_trigger": false}
]
```

查询必须真实，像是 Claude Code 或 Claude.ai 用户会实际输入的内容。不要抽象请求，而要具体、细节充分。例如文件路径、与工作或个人情境相关的背景、列名与取值、公司名、URL、一点前因后果。有的可以全小写、含缩写或错别字、口语化。长度要有变化，并侧重边界情况而非一眼能判明的例子（用户仍有机会确认）。

差：`"Format this data"`、`"Extract text from PDF"`、`"Create a chart"`

好：`"ok so my boss just sent me this xlsx file (its in my downloads, called something like 'Q4 sales final FINAL v2.xlsx') and she wants me to add a column that shows the profit margin as a percentage. The revenue is in column C and costs are in column D i think"`

对**应触发**的查询（8～10 条），考虑覆盖面。同一意图用不同说法——有的正式、有的随意。包含用户未明说技能名或文件类型但明显需要该技能的情况。加入不常见用例，以及与另一技能竞争但应由本技能胜出的情况。

对**不应触发**的查询（8～10 条），最有价值的是「擦边球」——与本技能共享关键词或概念但实际需要别的东西。考虑相邻领域、歧义表述（朴素关键词匹配会误触但不应触发的情形），以及触及本技能所做之事但另一工具更合适的语境。

要避免：不应触发的查询不要明显无关。对 PDF 技能用「写斐波那契函数」作负例太简单——测不出东西。负例应真正棘手。

### 第二步：与用户审阅

用 HTML 模板向用户展示评测集供审阅：

1. 从 `assets/eval_review.html` 读取模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评测项的 JSON 数组（不要加引号包裹——它是 JS 变量赋值）
   - `__SKILL_NAME_PLACEHOLDER__` → 技能名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 技能当前描述
3. 写到临时文件（例如 `/tmp/eval_review_<skill-name>.html`）并打开：`open /tmp/eval_review_<skill-name>.html`
4. 用户可编辑查询、切换是否应触发、增删条目，然后点击「Export Eval Set」
5. 文件会下载到 `~/Downloads/eval_set.json`——若有多份，检查「下载」文件夹里最新的一份（例如 `eval_set (1).json`）

这一步很重要——差的评测查询会导致差的描述。

### 第三步：运行优化循环

告诉用户：「这会花一些时间——我会在后台运行优化循环并定期查看进度。」

把评测集保存到工作区，然后在后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

使用系统提示中当前会话所用的模型 ID，使触发测试与用户实际体验一致。

运行期间可定期 tail 输出，向用户汇报当前迭代轮次与分数概况。

这会完整自动跑优化循环：将评测集划分为 60% 训练、40% 留出测试；评估当前描述（每条查询跑 3 次以获得可靠触发率）；再调用 Claude 根据失败情况提出改进；对每条新描述在训练集与测试集上重新评估，最多迭代 5 次。结束后会在浏览器打开 HTML 报告展示各轮结果，并返回含 `best_description` 的 JSON——按测试集分数而非训练集选取，以减少过拟合。

### 技能触发如何运作

理解触发机制有助于设计更好的评测查询。技能以名称 + 描述出现在 Claude 的 `available_skills` 列表中，Claude 根据描述决定是否查阅技能。重要的是：Claude 只在自身不太能轻松处理任务时才会查阅技能——简单一步的查询如「读这个 PDF」即使描述完全匹配也可能不触发技能，因为 Claude 可用基础工具直接处理。复杂、多步骤或专业化的查询在描述匹配时更可靠地触发技能。

因此评测查询应足够有分量，使 Claude 确实能从查阅技能中受益。像「读文件 X」这类简单查询是差的测试用例——无论描述好坏都可能不触发技能。

### 第四步：应用结果

从 JSON 输出中取 `best_description`，更新技能的 SKILL.md 前置元数据。向用户展示前后对比并汇报分数。

### 打包与呈现（仅当存在 `present_files` 工具时）

检查是否有 `present_files` 工具。若没有则跳过本步。若有，打包技能并向用户呈现 `.skill` 文件：

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

打包后告知用户生成的 `.skill` 文件路径，以便安装。

## Claude.ai 专用说明

在 Claude.ai 上，核心流程相同（草稿 → 测试 → 审阅 → 改进 → 重复），但因没有子代理，部分机制需调整。适配要点如下：

**运行测试用例**：无子代理即无并行。对每个测试用例，阅读技能的 SKILL.md，再按其说明亲自完成测试提示词。一次一个。这比独立子代理不够严格（技能是你写的、运行也是你，你掌握全部上下文），但作为健全检查仍有价值——人工审阅会弥补。跳过基线运行——只用技能按请求完成任务。

**审阅结果**：若无法打开浏览器（例如 Claude.ai 的虚拟机无显示器，或你在远程服务器上），完全跳过浏览器审阅。改为在对话中直接展示结果。对每个测试用例展示提示词与输出。若输出是用户需要查看的文件（如 .docx、.xlsx），保存到文件系统并告知路径以便下载检查。内联征求反馈：「看起来如何？有没有想改的？」

**基准测试**：跳过定量基准——它依赖基线对比，没有子代理时意义不大。以用户的定性反馈为主。

**迭代循环**：与之前相同——改进技能、重跑测试用例、征求反馈——只是中间没有浏览器审阅。若仍有文件系统，仍可将结果整理到各轮迭代目录。

**描述优化**：本节依赖 `claude` CLI（具体为 `claude -p`），仅在 Claude Code 中可用。若在 Claude.ai 上则跳过。

**盲测对比**：需要子代理。跳过。

**打包**：`package_skill.py` 在任何有 Python 与文件系统的环境都可运行。在 Claude.ai 上可运行，用户可下载生成的 `.skill` 文件。

**更新已有技能**：用户可能是在更新已有技能，而非从零创建。此时：

- **保留原始名称。** 注意技能目录名与 `name` 前置字段——保持不变。例如已安装技能为 `research-helper`，应输出 `research-helper.skill`（不要 `research-helper-v2`）。
- **编辑前先复制到可写位置。** 已安装路径可能只读。复制到 `/tmp/skill-name/`，在那里编辑并打包。
- **若手动打包，先在 `/tmp/` 暂存**，再复制到输出目录——直接写入可能因权限失败。

## Cowork 专用说明

若在 Cowork 中，主要须知：

- 有子代理，因此主流程（并行派生测试用例、跑基线、评分等）均可使用。（但若严重超时，串行跑测试提示词也可以。）
- 无浏览器或显示器，生成 eval 查看器时请使用 `--static <output_path>` 写独立 HTML，而不是启动服务器。然后提供用户可在浏览器中打开的链接。
- 不知为何 Cowork 环境似乎容易让 Claude 在跑完测试后不去生成 eval 查看器，故再强调一遍：无论在 Cowork 还是 Claude Code，跑完测试后应**始终**用 `generate_review.py` 为人类生成 eval 查看器，让人在看例子之后再由你自行评估输入并试图修正——**不要**手写 HTML。抱歉这里用大写强调：在**你自己**评估输入**之前**，先生成 EVAL 查看器。要尽快把结果摆到人面前！
- 反馈方式不同：没有常驻服务器时，查看器的「Submit All Reviews」会把 `feedback.json` 作为文件下载。你可从该处读取（有时需先请求访问权限）。
- 打包可用——`package_skill.py` 只需 Python 与文件系统。
- 描述优化（`run_loop.py` / `run_eval.py`）在 Cowork 中应可正常工作，因其通过 subprocess 调用 `claude -p` 而非浏览器，但请放在技能与用户都确认满意之后再跑。
- **更新已有技能**：用户可能是在更新已有技能。遵循上文 Claude.ai 一节中的更新指引。
